# Android 迁移背景

这份文档说明：为什么要把 Android 客户端从 `Xray-core-laochendeai` 中单独拆成新仓库，以及新仓库应该继承什么，不应该机械照搬什么。

## 为什么单独建仓

核心仓库的职责已经很明确：

- 保留 Xray-core runtime
- 维护内嵌 WebPanel
- 围绕本机透明代理、TUN、DNS 分流、节点池和订阅管理做本地控制面

但它当前对 Android 的交付仍然停留在“核心二进制产物”层面，而不是“可安装客户端”层面：

- release 里的 Android 产物是 `zip`，里面是 CLI binary，不是 APK
- 当前没有正式 Android app 交付链路
- 当前也没有 Android 32 位发布产物

因此 Android 客户端要继续做下去，必须有单独仓库承接：

- Android 工程结构
- APK / AAB 打包
- 移动端 UI / 交互
- 手机上的运行时控制
- 与核心仓库不同的发布节奏

## 从核心仓库继承什么

新 Android 仓库不该只是“换个壳”。真正要继承的是核心仓库已经验证过的产品语义。

### 1. 订阅导入语义

Android 客户端应继承以下导入能力：

- 远程 URL 导入
- 手工粘贴导入
- 本地文件导入
- 后续补齐二维码扫码导入

并保持以下约束：

- 删除订阅时，不应该粗暴删除全部节点历史
- 订阅里临时消失的节点，应该回到生命周期管理，而不是直接消失

### 2. 节点池生命周期

核心仓库的节点池语义值得直接继承：

- 候选
- 验证中
- 活跃
- 隔离
- 已移除

Android 端后续也应保持：

- 默认不让未验证节点直接承担真实流量
- 只有活跃池可以进入自动选择或透明代理候选
- 已移除节点可以恢复，而不是只能永久删除

### 3. 稳定性优先的控制面语义

核心仓库当前已经把“稳定优先”明确成第一阶段原则，Android 端应继承这条主线：

- 活跃池不健康时，不能继续让流量跑在不可信节点上
- 当运行态退化时，应优先回到干净状态
- 启动后要能回到可解释、可恢复的默认状态
- 节点干净度、网络类型、失败原因等状态要能被操作者直接看到

### 4. 操作入口布局思想

核心仓库已经把一些产品决策做清楚了，Android 端可以直接复用：

- 节点池是主要操作面
- 诊断页只保留诊断，不承担主操作流程
- 与透明代理有关的策略、白名单、远端 DNS、节点选择策略，应尽量在同一工作区完成

## 不应该直接照搬什么

Android 端不应把当前 WebPanel 逐页原样翻译成 App。

不要直接照搬：

- 桌面式多页面布局
- WebPanel 的所有诊断字段
- 为鼠标/宽屏设计的交互层级
- 仅适合本机 root/helper 的 Linux 实现细节

应该保留的是语义，而不是表现形式。

## Android 仓库当前边界

当前新仓库已经具备：

- 独立 Git 仓库
- 最小 Android 工程
- 本地 debug APK 构建
- GitHub Actions debug APK artifact

当前还没有：

- Xray core 集成
- 前台服务 / VPN Service
- 真正的移动端透明代理
- 扫码订阅
- 节点池完整 UI
- 目标站点绑定

## 参考项目

这些项目值得在后续实现时按需参考，但不应盲目复制：

- `v2rayNG`
- `X-flutter`
- `XTLS/libXray`
- `AndroidLibXrayLite`

它们可以提供：

- Android 端核心集成方式
- 订阅和 URI 解析样例
- VPN / Service 生命周期参考
- 分享链接、扫码导入等移动端交互参考

## 这份文档的用途

后续开 issue 时，先判断该需求属于哪一种：

- Android 仓库应该新增的移动端能力
- 仍然应该留在核心仓库的 runtime / helper 能力
- 两边都要做，但边界必须分清

如果需求跨越两边仓库，不要在一个仓库里假装全做完。
